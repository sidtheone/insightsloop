## Values Alignment

### "Delete before you add."

The trimmed plan earns this. The bloated version had 15 sections. This has 9. Init, theme gap fill, forced uncertainty, triage formula — all cut. The plan practices what the value preaches.

**One tension:** §9 (Condense SKILL.md) proposes deleting 520 lines from devloop SKILL.md. Deletion is good. But the plan also adds 10 new template files + edits to 7 SKILL.md files. Net file count goes up by 10. Net complexity arguably goes up (10 template files + 7 SKILL.md edits is more surface area than 1 long SKILL.md). The argument is that the 10 templates are simpler than the prose they replace — and they are. But this is addition disguised as deletion. The value should be satisfied by asking: "Could we ship with fewer templates?" For Lean crew (Small features), only 5 templates are used (quartermaster, sentinel, shipwright, storm-tdd, storm-verify). The other 5 (quartermaster-correction, monkey-frame, monkey-build, shipwright-scaffold, cartographer) are Full-crew or correction-mode only. Ship the 5 that matter for every run. Add the rest when Full crew or correction mode is validated on a real run.

### "Three lines beat a clever abstraction."

Brief templates are an abstraction layer. The alternative (prose instructions telling the orchestrator what to paste) was the current system, and it failed — context death in v0.11. The abstraction earns its place because the problem is measured, not theoretical. This value is satisfied.

Impact tags (§7) are three words added to Monkey output: `breaks-build`, `values-gap`, `nice-to-have`. That's the three-line solution to the "orchestrator interprets findings" problem. Clean.

Summary returns (§2) are ~8 lines of format specification. The alternative was full agent output in orchestrator context (~200+ lines per agent). The format is simple and the problem is real. Clean.

### "Read it top to bottom or rewrite it."

The symbolic notation in §9 is meant to improve readability. Does it? The current 682-line SKILL.md is not readable top to bottom — too much inline brief content, repeated patterns, verbose error handling. The symbolic notation compresses this but introduces a different readability challenge: symbols are only readable if you already know the system. `Sentinel(template) → test files` is clear to someone who's read the SKILL.md before. For a new reader, it's opaque. The plan mitigates this by keeping crew table, definitions, and scope rules as prose. The symbolic parts are the flow — which is the part that benefits most from compression.

**Concern:** The fix pipeline in symbolic notation (`Storm(spec) → Sentinel(test) → Shipwright(patch)`) omits the brief construction step. A reader seeing this doesn't know what context each agent receives. The current SKILL.md spells this out. The templates are supposed to replace the inline content, but there are no fix-pipeline templates (see #9).

### "Untested code doesn't leave the engine."

The plan adds no tests for the engine itself. Brief templates, summary returns, resume detection, impact tags — all are behavioral rules in SKILL.md files, not tested code. This is a pre-existing gap (the orchestrator is untested) that the plan neither widens nor narrows. Neutral.

### "Subtract until it breaks, then add one back."

The trimmed plan is the result of three rounds of subtraction. 15 sections became 9. 1006 lines became 220. What survived the cuts: brief templates (fixes context death), summary returns (fixes context death), resume (user-requested), Lean crew (simplifies small features), CLI deps (prevents version drift), plan correction (prevents L-001 violation), impact tags (prevents orchestrator interpretation), Sentinel gate (prevents runtime failures), and the condensed SKILL.md (reduces reading load).

Could more be cut? Resume (§3) is 3 lines and covers a real edge case (v0.11 died mid-run). Lean crew (§4) removes agents, which is subtraction. CLI deps (§5) is one rule. Plan correction (§6) is one Quartermaster mode addition that prevents the orchestrator from doing planning work. Impact tags (§7) is one Monkey output field. Sentinel gate (§8) is 3 checks. None of these are fat.

The only section where "subtract until it breaks" might apply: §9 (Condense SKILL.md). The plan proposes a full rewrite. A full rewrite is risky — you lose institutional knowledge baked into phrasing choices. An alternative: apply §1-§8 first, then see if the SKILL.md is still too long. The condensation might be unnecessary after templates move inline content out.

## Summary

**11 introduced issues:** 4 high, 5 medium, 2 low
**5 consistency findings:** 4 actionable, 1 no action needed
**3 pre-existing issues:** 2 medium, 1 low

### Top 5 — Highest Impact

1. **#1 (high) — Who writes the artifact file?** Summary returns only work if agents write their own artifacts before returning. Current SKILL.md assumes the orchestrator writes artifacts from agent output. If you ship summary returns without changing artifact ownership, the orchestrator receives 25 lines and has no full content to write. This is the load-bearing question of §2.

2. **#9 (high) — Fix pipeline has no templates.** The symbolic notation in §9 references `Storm(spec) → Sentinel(test) → Shipwright(patch)` but §1 defines no templates for these. Either add 3 fix-pipeline templates or explicitly note they're inline. Without this, the condensed SKILL.md has a gap where three agent invocations have no brief construction path.

3. **#10 (high) — Theme compression loses exclusion constraints.** Compressing the theme system to one line ("adopt that voice") drops 5 explicit "never theme this" rules. The orchestrator will theme crew output, user gates, and artifact content unless told not to. These are behavioral constraints, not obvious defaults.

4. **#5 (high) — "Peer dep warnings? STOP" is too aggressive.** React 19 + most React ecosystem libraries produce peer dep warnings. This rule will stop the Shipwright on nearly every React project. Distinguish warnings (log and continue) from errors (stop).

5. **C1 (consistency) — Two categorization schemes at convergence.** Monkey uses impact tags, Storm uses severity. The convergence gate mixes both. Add a normalization step at consolidation or use one scheme everywhere.

### What the Trimmed Plan Got Right

- **Cutting Init, theme gap fill, forced uncertainty, triage formula.** Each was an abstraction the engine doesn't need yet. "Delete before you add" in action.
- **Monkey self-tags instead of orchestrator categorizing.** Fixes L-001 at the source. Three words (`breaks-build`, `values-gap`, `nice-to-have`) instead of a categorization system.
- **Resume as 3 lines.** The bloated version had a 7-row artifact lookup table, staleness detection, and inconsistency handling. This version says "if artifacts exist, ask." Simpler. Loses granularity, gains maintainability.
- **"No slot resolution table — the template is the spec."** The template files are the single source of truth. No table to get out of sync. The tradeoff (ambiguous slot names) is manageable with naming discipline.
- **Section count: 9.** Every section either fixes a measured failure (§1, §2 fix context death), removes complexity (§4, §9), or adds a single rule to prevent a known issue (§5, §6, §7, §8). Nothing is speculative.
